### QA-004 — 宣稱可上傳 20MB，實際約 7.5MB 就失敗且錯誤訊息錯誤

- 嚴重度：P1
- 狀態：已修復，live 邊界回歸通過

- 重現：以合法純文字檔依序測試 7.25MB、7.5MB、8MB、20MB。
- 實際結果：7.25MB 回 200；7.5MB 起穩定回 400 `缺少檔案內容`。route 內宣告的 20MB `MAX_BYTES` 尚未執行，JSON/base64 request 已在更前面無法解析。
- 預期結果：產品標示的 20MB 應真能上傳；超過限制應回 413 與「檔案太大」，不能謊稱沒有檔案。

- 回歸：剛好 20MB 回 200 且完成抽取／保存；20MB+1 byte 回 413，訊息正確。

### QA-005 — 非法 base64 的錯誤分類不正確

- 嚴重度：P3
- 狀態：已修復，live API 回歸通過
- 範圍：`POST /api/extract-text`
- 重現：送 `dataBase64="%%%NOT-BASE64%%%"`。
- 實際結果：回 422，宣稱它是無法解析的 `.txt` 二進位檔；`Buffer.from(..., "base64")` 不會因非法字元丟錯，因此現有 try/catch 無法驗證編碼。

### QA-007 — 匯入檔仍可不用 custom-code 讀取並外送任意本機檔案

- 嚴重度：P0（安全）
- 狀態：已修復，live 信任閘門 E2E 通過
- 範圍：workflow 匯入、`read-file`、`http-request` 等既有節點組合
- 現況：匯入時會正確清空頂層及 repeat-steps 內的 custom-code，也會清空 send-email 收件人；但外來流程仍可設定 `read-file.path=/Users/.../敏感檔`，再用 `http-request POST` 把 `{{fileText}}` 送往任意公網網址。全程不需要 custom-code，現有 sanitizer 不會改任何一項。
- 預期結果：外來流程在第一次執行前必須有確定性的危險能力審核／確認，或匯入時清除所有本機路徑與外送目的地；不能只防 custom-code。
- 驗證限制：為避免真的外送使用者資料，本輪只確認匯入與節點執行能力，不向外部攻擊站點發送資料。
- 修復：所有外來 workflow 標記 `importedUntrusted`；第一次執行一律回 409，UI 明確說明本機讀檔／網路外送風險並要求人為確認。確認後 server 重新讀最新版、原子保存信任狀態才允許執行；引擎層也有第二道阻擋，避免其他觸發入口繞過。
- 回歸：安全匯入首次執行 409、確認後 200 且之後狀態可信；外來 code-only custom-code 在匯入時因清空後無 intent 直接 400。

### QA-014 — 複雜建圖因無歧義的欄位型別別名浪費整輪模型呼叫

- 嚴重度：P1（效能／可用性）
- 狀態：已修復，確定性正規化單元回歸通過
